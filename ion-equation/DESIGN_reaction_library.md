# 反応ライブラリとインデックス設計（2026-07-24 合意）

「反応式を網羅し、検索できるライブラリにする」ための設計書。
既存の [DESIGN_reaction_types.md](DESIGN_reaction_types.md)（反応3群とアニメ分類）と
[DESIGN_redox.md](DESIGN_redox.md)（酸化還元エンジン）の上位に立ち、データ構造・分類・検索・
アニメーションタイプの管理方針を定める。**進行状況は DEVELOPMENT.md が正**。

## 目的

1. 反応式を**網羅**する（中和・沈殿・気体・酸化還元・錯イオン…）
2. **分類から検索**（反応の型・塩の分類・酸化還元の種別・単元）
3. **登場する物質から検索**（「KMnO₄ を含む反応」「Fe²⁺ が出る反応」）
4. **分類に応じたアニメーションタイプ**を順次実装する（未実装が多い前提の設計）
5. 酸化還元を**溶液中の反応（KMnO₄ など）**に拡張する（金属樹だけでない）

## 設計の核：ライブラリ層とアニメ層を分離する

網羅（データ）とアニメ（実装）を切り離すのが要。

- **反応エントリ（データ）** — 反応式・イオン反応式・登場物質・分類・単元・`animationType`・解説。
  これは**先に全部そろえられる**。
- **アニメーションタイプ（実装）** — 分類に紐づく描画エンジン。`animationType` が実装済みなら
  「遊べるステージ」、未実装なら**インデックスに式・解説だけ載る参照エントリ**（または汎用フォールバック）。

これにより「まだ作っていないアニメが多い」問題が解ける：反応は先に網羅してインデックスに載り、
アニメは分類ごとに後から足す。反応が `animationType` を宣言し、レジストリが type→描画を対応づけ、
未実装なら「準備中」表示にフォールバックする。

```
反応エントリ ──(animationType)──▶ アニメレジストリ ──▶ 実装あり: 遊べる
                                                    └─▶ 実装なし: 参照のみ（式＋解説）
```

## データアーキテクチャ：reactions.json へ移行

現行の `model.js` 埋め込み配列（STAGES / REDOX_STAGES）から、**`reactions.json` に外出しして
fetch する**方式へ移行する（網羅・検索・自動検証・大量作成に向く。姉妹 Chem-Assembler と同じ流儀）。

- **重要な帰結**：`fetch` 導入で **file:// 直開きは不可**になり、**ローカルサーバー必須**になる。
  DEVELOPMENT.md の設計原則「v1 は JS 埋め込み」の記述をこの移行時に更新すること。
  公開は既にサーバー経由（GitHub Pages）なので本番影響はない。
- `model.js` は**純ロジック（判定・集計・電離・半反応式演算）に専念**し、データは JSON に移す。
  SPECIES / STRUCTURE / PARTS / DISSOCIATION も JSON 化を検討（種の網羅と検証がしやすくなる）。
- 派生インデックス（**物質→反応の逆引き**、分類→反応の一覧）は**ロード時にコードで構築**する
  （JSON には持たせない。`species` 配列から自動生成）。

### reactions.json スキーマ（案）

```jsonc
{
  "id": "neutral-h2so4-2naoh",              // 一意キー（kebab）
  "reactants": ["H2SO4", "NaOH"],
  "products":  ["Na2SO4", "H2O"],
  "coeffs":    [1, 2, 1, 2],                // reactants→products の順。最簡整数比
  "netIonic":  "H⁺ + OH⁻ → H₂O",
  "classes": {                              // 分類（複数軸。検索の主キー）
    "type": "中和",                          // 反応の型（enum、下記タキソノミー）
    "saltType": "正塩",                      // 塩の分類（該当時のみ）
    "redox": null                           // 酸化還元の種別（該当時のみ）
  },
  "units": ["中和"],                         // 単元タグ（教科対応。複数可）
  "species": ["H2SO4","NaOH","Na2SO4","H2O","H+","OH-","SO4^2-","Na+"], // 検索用（分子＋イオン）
  "animationType": "combine",               // アニメレジストリのキー
  "rules": [ { "find":["H+","OH-"], "make":"H2O", "kind":"combine" } ], // アニメ駆動データ（型依存）
  "note": "強酸×強塩基の完全中和。傍観イオン Na⁺・SO₄²⁻ が残る。",
  "difficulty": 1,                          // 1〜5（表示・絞り込み用）
  "playable": true                          // false＝参照のみ（animationType 未実装 or 出題化しない）
}
```

- **`species`** は検索の要。分子と、水中で生じる主なイオンの両方を入れる（逆引きの網羅性のため）。
  ロード時に `species → [reactionId...]` の逆引きマップを構築する。
- `rules` / 半反応式など**アニメ駆動データは型ごとに形が違う**。共通フィールドは上記、
  型固有フィールドは `animationType` ごとに定義（レジストリが解釈）。
- `playable:false` はインデックスに出るが遊べない（式・イオン反応式・解説・分類のみ表示）。

## 分類タキソノミー（enum）

検索フィルタの軸。**反応の型**を主軸に、補助軸で塩・酸化還元・単元を持つ。

### 反応の型 `classes.type`
`中和` / `沈殿` / `気体発生` / `弱酸弱塩基の遊離` / `酸化還元` / `錯イオン生成` /
`加水分解` / `分子反応`（化合・分解・燃焼） / `その他`

### 塩の分類 `classes.saltType`（中和・塩系のみ）
`正塩` / `酸性塩` / `塩基性塩`

### 酸化還元の種別 `classes.redox`（酸化還元のみ）
`金属の析出`（イオン化傾向・金属樹） / `金属と酸`（H₂ 発生） /
`溶液中の酸化剤還元剤`（KMnO₄・K₂Cr₂O₇ など） / `ハロゲンの酸化力` / `電池` / `電気分解`

### 単元タグ `units`（教科対応。ステージ横断）
`中和` / `中和滴定` / `塩の分類` / `沈殿反応` / `気体の製法` / `イオン化傾向` /
`酸化還元滴定` / `電池・電気分解` / `錯イオン` …（現行 STAGE_TAGS を発展）

## 検索・インデックス UI（案）

トップに**反応インデックス**画面を置く。

- **分類フィルタ**：型・塩の分類・酸化還元種別・単元・難易度をチップで AND/OR 絞り込み
- **物質検索**：入力（例「KMnO4」「Fe2+」）→ `species` 逆引きで該当反応を列挙
- 一覧の各行：反応式（係数つき）＋分類バッジ＋`playable` なら「遊ぶ」ボタン、
  参照のみなら式・イオン反応式・解説を展開
- 反応詳細はディープリンク可能に（`?rxn=<id>`）— 授業・共有で個別反応を指せる（要検討）

## アニメーションタイプ・レジストリ

各反応が `animationType` を宣言し、レジストリ `ANIMATIONS[type] = 描画エンジン` が対応づける。
未登録・未実装は**「準備中」フォールバック**（式・イオン反応式・数合わせビューだけは共通で出せる）。

| 分類 | animationType | 駆動データ | 状態 |
|---|---|---|---|
| 中和 | `combine` | rules(find/make, kind=combine) | ✅ 実装済み |
| 沈殿 | `precipitate` | rules(kind=precipitate) | ✅ |
| 気体発生 | `gas` | rules(kind=gas, via) | ✅ |
| 酸性塩・部分中和 | `acid-salt` | rules + saltGoal(ions) | ✅ |
| 酸化還元・金属析出 | `redox-metal` | 半反応式×倍率＋板 | ✅（redox.js） |
| **酸化還元・溶液中** | `redox-solution` | 半反応式＋色変化＋H⁺ | ❌ **要開発（次の主役）** |
| 錯イオン生成 | `complex-ion` | 配位子付加・色変化・沈殿再溶解 | ❌ |
| 弱酸弱塩基 | `weak-partial` | 部分電離・平衡（往復） | ❌（M4） |
| 分子反応（C群） | `molecular` | 原子ばらけ→再結合 | ❌ |

- 下段の**数合わせビュー**は全型共通基盤（PARTS の差し替えで対応、DESIGN_reaction_types.md 参照）。
  参照のみの反応でも係数練習は成立させられる。

## 酸化還元の溶液中への拡張（redox-solution）

現行 redox は「金属板＋溶液イオン」前提。溶液中どうしの電子授受は**新アニメ**が要るが、
**モデルは既存の `HALF_REACTIONS` 拡張で足りる**（半反応式×倍率→e⁻ 打ち消し→イオン反応式、の枠に乗る）。

### 追加する半反応式（例）

酸化剤（e⁻ を受け取る・酸性条件で H⁺ と水が絡む）:
- `MnO₄⁻ + 8H⁺ + 5e⁻ → Mn²⁺ + 4H₂O`（**紫 → ほぼ無色/淡桃**）
- `Cr₂O₇²⁻ + 14H⁺ + 6e⁻ → 2Cr³⁺ + 7H₂O`（**橙 → 緑**）
- `H₂O₂ + 2H⁺ + 2e⁻ → 2H₂O`（酸化剤として働く場合）

還元剤（e⁻ を出す）:
- `Fe²⁺ → Fe³⁺ + e⁻`
- `(COO)₂²⁻（シュウ酸イオン） → 2CO₂ + 2e⁻`
- `H₂O₂ → O₂ + 2H⁺ + 2e⁻`（還元剤として働く場合＝両性）
- `2I⁻ → I₂ + 2e⁻`

代表反応（イオン反応式）:
- `MnO₄⁻ + 5Fe²⁺ + 8H⁺ → Mn²⁺ + 5Fe³⁺ + 4H₂O`
- `2MnO₄⁻ + 5(COO)₂²⁻ + 16H⁺ → 2Mn²⁺ + 10CO₂ + 8H₂O`

### 新アニメの見せどころ
- **色変化**：溶液全体（または粒）の色を反応進行で遷移（紫→無色）。滴定の終点＝
  「紫が消えなくなる」を演出できる → 単元 `酸化還元滴定` に直結
- **H⁺ の消費・H₂O の生成**：酸性条件が反応に必須なことを可視化
- **板なしの電子授受**：溶液中の粒どうしで e⁻ が渡る（`redox-metal` の板を外した動き）
- モデル不変条件（既存テストと同型）：酸化数の合計＝電荷、Δ酸化数＝e⁻ 数、
  半反応式の原子・電荷保存、e⁻ 打ち消し後のイオン反応式がつり合う

### 酸化数の拡張が必要な種
`OXIDATION` 表に Mn（+7/+2）、Cr（+6/+3）、O（過酸化物 −1）、C（シュウ酸/CO₂）などを追加。
多原子イオン内の酸化数（MnO₄⁻ の Mn=+7 は O=−2 から逆算）を扱う規則を DESIGN_redox.md に追記する。

## 段階計画（進行は DEVELOPMENT.md が正）

- **Phase 0：設計（本書）** — スキーマ・分類・レジストリ・移行方針を確定 ← いまここ
- **Phase 1：データ化と検証** — reactions.json 定義、既存12＋主要反応を移行、
  機械検証（原子・電荷保存、`classes`↔`rules` 整合、`animationType` 実在、`species` 網羅）。
  ローダを model.js/app.js に追加（サーバー必須化を DEVELOPMENT.md に反映）
- **Phase 2：インデックス/検索 UI** — 分類フィルタ＋物質逆引き。参照エントリの表示形式
- **Phase 3：アニメレジストリ整理** — 既存 combine/precipitate/gas/acid-salt/redox-metal を
  `animationType` に整理、未実装は「準備中」フォールバック
- **Phase 4：redox-solution 実装** — KMnO₄ を先頭に、色変化・H⁺参加の新アニメ
- **Phase 5 以降：** 錯イオン生成 / 弱酸弱塩基（M4）/ 分子反応（C群）を順次

## 品質保証（reactions.json の機械検証）

コミット前テストに追加する検証（データ駆動なので反応を足すほど自動で網羅が広がる）:

1. 全反応で**原子と電荷が保存**（coeffs で左右一致、最簡整数比）
2. `netIonic` が原子・電荷保存（傍観イオン除去後）
3. `species` が反応の全登場種（分子＋主要イオン）を**漏れなく**含む（逆引きの正しさ）
4. `classes.type` / `saltType` / `redox` / `units` が**タキソノミーの enum 内**
5. `animationType` が**レジストリに存在**（未実装なら「準備中」扱いだが型名は既知）
6. `rules` / 半反応式などアニメ駆動データが**原子・電荷保存**（型ごとの検証）
7. `id` の一意性。`playable:true` は既存の総なめ UI テスト対象に自動編入

## 未決事項（実装時に確定する）

- **難易度/学年**の粒度（1〜5か、中学/高基礎/高発展か）
- **参照のみ反応**の表示形式（式＋イオン反応式＋解説のみ／数合わせは出すか）
- 反応の**ディープリンク**（`?rxn=<id>`）を入れるか
- SPECIES/STRUCTURE/PARTS/DISSOCIATION も JSON 化するか（Phase 1 で範囲を決める）
- 既存 STAGES / REDOX_STAGES の**統合順序**（一気に移すか、両立期間を置くか）
- redox-solution の**色**の表現（溶液全体を塗るか、粒の色遷移か、両方か）
