# Chem-Assembler 開発方針

本ドキュメントは、Chem-Assembler（有機化学構造パズル）の開発における設計原則・コーディング規約・品質保証手順・作業ロードマップを定める。以後の修正・機能追加はすべて本方針に従って行う。

---

## 1. プロジェクトの目的と開発コンセプト

- 高校〜大学初級の有機化学学習者が、分子構造（トポロジー・結合次数・不斉炭素）を手で組み立てて理解するためのブラウザゲーム。
- **「化学的に正しい答えは必ず正解になる」ことを最優先の品質基準とする。** 誤検証（正しいのに不正解）はゲームとして最も致命的な欠陥であり、他のあらゆる作業に優先して修正する。

### 1.1 開発コンセプト（作図思想）

本アプリは **立体模型の忠実再現ではなく、紙に手書きするのと同じ感覚での構造式作図** を目指す。

1. **手書き感覚の優先**: 学習者がノートに構造式を書くときの感覚（直線・直角中心の作図）を再現する。例えばアルカンは、実際の結合角（109.5°）を反映したジグザグの折れ線ではなく、**直交方向（上下左右90°）に作図するのが本アプリの正**である。
   - したがって、直交グリッドスナップ・90°方向の水素配置は「不正確さ」ではなく**意図された仕様**であり、リアルな結合角に近づける方向の「修正」を行ってはならない（過去にsp3自動レイアウトがドリフトバグを生んで撤去された経緯がある）。
   - 例外として、構造式の慣習に従い sp2（C=C、ベンゼン環）は120°、sp（C≡C）は180°の作図を用いる。これは紙の上の構造式でも同様に書かれるためである。
2. **反応機構アニメーションへの発展**: 将来的に、巻矢印（曲がった矢印）による電子の動き・結合の組み換えをアニメーションで再現し、反応機構の学習をサポートすることを目指す。データモデル（原子・結合のグラフ構造）はこの発展を妨げない形を保つ。
3. **立体模型との対照**: 作図はあくまで平面の構造式としつつ、「作図上ではこう書くが、実際の分子はこうなっている」という対照提示（立体表現の併記・比較ビュー等）ができるとなおよい。平面作図の正しさと立体的な実像を混同させない見せ方を将来検討する。

## 2. アーキテクチャ原則

| ファイル | 責務 | 禁止事項 |
|---|---|---|
| `chemistry.js` | 化学モデル（原子・結合・分子・価標・同型判定・不斉判定）。**DOMに依存しない純粋ロジック** | DOM操作・描画・イベント処理を書かない |
| `game.js` | UI・描画（SVG）・入力処理・ステージ進行 | 化学的な判定ルールを直接実装しない（chemistry.jsに委譲） |
| `stages.json` | 問題データ（純粋データ、コメント不可） | ロジックを持ち込まない |
| `index.html` / `style.css` | 構造とスタイル | インラインstyleの新規追加は避け、style.cssに寄せる |

- 外部ライブラリ・ビルドツールは導入しない（vanilla JS + 静的配信を維持。GitHub Pages 互換）。
- 定数は一元管理する。価標は `VALENCIES`（chemistry.js）を唯一の情報源とし、重複定義（例: 旧 `getMaxBondType` 内のswitch）を作らない。

## 3. コーディング規約

### 3.1 文字エンコーディング（最重要）

- **すべてのファイルは UTF-8（BOMなし）で保存する。** 過去に Shift-JIS との二重変換でコメント・UI文字列が広範囲に文字化けした事故があるため、以下を徹底する:
  - コミット前に、変更ファイルに文字化けパターン（`縺`・`繧`・`繝`・`蜊`・`荳`・`邨`・`蛾`・`倡` 等の頻出化けクラスタ）が含まれていないか確認する。v104でこのパターン外の化け（`蜊倡ｵ仙粋`＝単結合）を発見したため検査文字を拡充した。
  - ユーザーに表示される日本語文字列（alert/confirm/textContent/innerHTML）を変更した場合は、必ずブラウザで実表示を確認する。
- 既存の文字化けコメントは、修正対象ファイルに触れる際に随時正しい日本語へ復元するか削除する。

### 3.2 スタイル

- 命名・コメント密度は既存コードに合わせる（コメントは日本語）。
- デッドコードは残さない。「無効化した関数」を空実装で残す場合は、呼び出し側ごと削除するか、残す理由をコメントで明記する。
- `substr` 等の非推奨APIは新規コードで使わない（既存は触るついでに置換）。

### 3.3 座標変換の原則

- **クライアント座標 ⇔ SVG座標の変換は必ず `svg.getScreenCTM().inverse()`（または `createSVGPoint().matrixTransform`）で行う。** `viewBox.width / rect.width` のような手計算は、`preserveAspectRatio` によるレターボックスを考慮できずズレを生むため禁止する。
- viewBox の操作（ズーム・パン・フィット）は既存方式を維持してよいが、座標の読み取りは上記に統一する。

### 3.4 入力処理の原則

- マウス・タッチの入力は **Pointer Events（`pointerdown`/`pointermove`/`pointerup`/`pointercancel`）に統一**することを目標とする。移行完了までの暫定対応でも:
  - タッチ系ハンドラでフェイクイベントを作る場合、**必ず座標（`changedTouches[0]` 等）を含める。** 座標なしイベントを座標依存の処理に渡さない。
  - 合成マウスイベントの二重発火を防ぐため、タッチを処理したら `preventDefault()` する。
- `alert` / `confirm` / `prompt` は原則使わない。通知は `#verify-result` のトースト表示、確認・入力はモーダルで行う。（既存の `window.onerror` の alert はデバッグ残骸であり撤去対象。）

### 3.5 Undo の原則

- `saveState()` は **実際に状態が変化する操作の直前のみ** 呼ぶ。空振り操作（何も消えない消しゴムクリック等）で履歴を消費しない。
- 状態の保存・復元は **原子の全プロパティ**（`isAsymmetricMarked`、`benzeneCenter`、`benzeneAngle` を含む）を対象とする。復元漏れを防ぐため、ホワイトリスト式のプロパティコピーではなく、シリアライズしたオブジェクトからの機械的復元を優先する。

## 4. 化学モデルの原則

1. **正解データ（stages.json）は、アプリ自身の価標ルールで構築可能でなければならない。** ユーザーが通常操作（原子配置・結合ツール・モジュール・結合次数トグル）で到達できない正解データを置かない。
2. **モジュール（官能基ボタン）が生成する構造は、そのモジュールを想定するステージの正解データと結合次数まで一致させる。** （例: -NO₂ モジュールは N(=O)(–O) を生成する。N(=O)(=O) は価標超過であり禁止。）
3. **芳香環のケクレ位相（単結合・二重結合の位置）の違いは正誤に影響させない。** 6員環で単・二重結合が交互になっている環は「芳香環」として扱い、検証時に環内結合の次数差を吸収する。
4. **二重結合の幾何（シス/トランス）は例外的に座標から読み取る。** 座標は原則「見た目専用」だが、C=C まわりの置換基の同側/反対側は2D構造式が幾何異性を伝える標準的な手段であるため、**命名表示（および将来の幾何指定ステージ）に限り**座標を参照する。対象は「環に含まれないC=Cで、両端の炭素がそれぞれ重原子置換基をちょうど1個持つもの」。直線に描かれた場合は「未指定」として扱い、通常のステージ正誤判定は引き続きトポロジーのみで行う。
5. 価標は `VALENCIES` を唯一の情報源とする。多価元素（S: 2/4/6 等）への拡張を行う場合は、モデル・検証・自動水素補完の三者を同時に対応させる。それまでは S を使う出題はスルホ基（6価）に限定する。
6. 自動水素補完は「描画とH数検証の一貫性」を守る。ユーザーがHを明示配置する機能は追加しない（現行仕様の維持）。

## 5. UI/UX の原則

### 5.0 作図スタイル（コンセプト1.1の具体化）

- スナップ・自動配置・自動水素は **直交（90°）を基本** とし、sp2は120°、spは180°とする。結合角の立体的リアリティを理由にこれを変えない。
- 検証ロジックは作図上の座標・角度に依存させない（トポロジーと結合次数のみで判定する現行方式を維持）。座標は見た目のためだけの情報とする。

- **ヘルプ・ヒントの記述と実装は常に一致させる。** 操作方法を変えたら同一コミットで `index.html` のヘルプを更新する。実装されていない操作をヘルプに書かない。
- 削除系の操作は影響を最小にする。**削除の影響は削除対象の原子・結合のみに限定し、連結が切れて孤立した原子を自動削除（巻き添え削除）しない。** 紙の上で消しゴムをかけたのと同じ挙動とする（コンセプト1.1-1）。孤立した原子が残ることは正常であり、ユーザーが自分で消すか再結合する。
- 判定結果メッセージは実際に検証した内容だけを述べる（不斉モードOFF時に「不斉炭素が一致」と表示しない）。
- 非表示中の機能（お手本表示・化学式・説明文）は、方針が決まるまで削除しない。ただし到達不能コードであることをコメントで明示する。

## 6. 品質保証（修正ごとの検証手順）

すべての修正は、コミット前に以下を実施する:

### 6.1 回帰テスト（必須）

1. ローカルサーバー（例: `python -m http.server 8123`）で起動し、コンソールエラーがないこと。
2. **`test.html` を開き、全テストが合格することを確認する。** 過去に修正した不具合（座標変換・ケクレ・削除・伸縮・振り分け・反応ビューア等）の再発を自動検出する回帰テストページであり、修正内容に関わらずコミット前に必ず実行する。
3. 新しい不具合を修正したときは、その再現ケースを `tests.js` にテストとして追加する。

### 6.2 回帰チェックリスト（該当領域を修正した場合）

- **座標系**: ウィンドウを縦長・横長にリサイズし、キャンバスの四隅付近でプレビュー（ゴースト）がカーソル直下に出ること。ズーム・パン後も同様。
- **検証ロジック**: 以下をブラウザコンソールで実行し、期待値になること。
  - ニトロベンゼン: -NO₂ モジュールで組んだ構造が正解になる。
  - o-キシレン: ベンゼン環モジュールの単結合側・二重結合側どちらにメチル基を付けても正解になる。
  - 酢酸: C-C-(=O)(-OH) が正解、C-C-(-O)(-OH)（次数違い）が不正解のまま。
- **Undo**: 不斉マーク → Undo → マークが復元される。配置 → Undo → 配置前に戻る。
- **削除**: プロパン中央Cの削除で両端のCが残る。
- **タッチ（対応時）**: タップ配置が1回で確定する（配置→即削除にならない）。タッチドラッグで結合が張れる。

### 6.3 検証の記録

- 各修正のコミットメッセージ本文に「何をどう検証したか」を1行以上書く。

## 7. バージョン管理・リリース手順

- コミットは **1修正=1コミット**。メッセージは既存の Conventional Commits 形式（`fix:` / `feat:` / `chore:`）を踏襲し、末尾に `(vNN)` を付ける。
- バージョン番号 `vNN` を上げる際は、**`index.html` と `test.html` の全キャッシュバスターを同一コミットで** 更新する（不一致はキャッシュ事故のもと。GitHub Pagesはmax-age=600で配信するため、バスターのないURLは最大10分間古い版が使われ続ける）:
  1. `index.html` の `style.css` / `chemistry.js` / `game.js` / `reaction.js` / `quiz.js` の `?v=NN`
  2. `index.html` ヘッダー表示 `<div class="version">vNN</div>`
  3. `test.html` の `index.html?v=NN`（iframe）と `tests.js?v=NN`
  4. `audit.html` の `index.html?v=NN`（iframe）と `audit.js?v=NN`（v108で追加）
  - 一括更新例: `sed -i 's/?v=OLD/?v=NEW/g' index.html test.html audit.html` ＋ ヘッダー表示の置換
- main ブランチ直コミット運用を維持する（単独開発のため）。ただし大きなリファクタ（Pointer Events 移行等）はブランチを切ってもよい。
