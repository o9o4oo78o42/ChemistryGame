# 反応実行モード 設計書（P9-1）

ユーザーが作図（または名称から呼び出し）した分子に対して「この分子の反応」を提示し、
反応を選ぶと生成物へ変化するモード。既存の反応機構ビューア（reaction.js＝登録済みデータの
観賞用）と異なり、**ユーザー自身の分子を対象に反応を実行する**のが本質。
実装前の合意用ドキュメントであり、実装の進行に合わせて更新する。

---

## 1. 体験イメージ

1. エタノールを作図（または名称呼び出しで配置）
2. 右パネルの「⚗ この分子の反応」カードに、分子のプロパティ（例: **1級アルコール**）と
   実行可能な反応（例: **酸化 → アセトアルデヒド**、**分子内脱水 → エテン**、
   **分子間脱水 → ジエチルエーテル**〔要2分子〕、**エステル化**〔要カルボン酸〕）が並ぶ
3. 「酸化」を選ぶと、キャンバス上の分子がアニメーションでアセトアルデヒドに変化。
   キャプションで「1級アルコールの酸化。さらに酸化すると酢酸になる」と解説
4. 続けて「酸化」を選べば酢酸へ（連鎖実行）。Undoでいつでも巻き戻せる
5. 反応表示は2段階を切替可能:
   - **簡易版（既定）**: 反応前 → 反応後のモーフィング（結合の組み換えをフェードで表現）
   - **機構版**: 対応する反応機構データ（reactions.json）がある場合、巻矢印アニメーションを再生

## 2. スコープ（対象反応・高校化学の定番）

| 分類 | ルール | 必要条件 | 生成物 |
|---|---|---|---|
| 酸化 | 1級アルコール→アルデヒド | C(-OH)にH2つ | R-CHO + (H₂O相当は省略) |
| 酸化 | アルデヒド→カルボン酸 | -CHO | R-COOH |
| 酸化 | 2級アルコール→ケトン | C(-OH)にH1つ | R-CO-R' |
| （案内） | 3級アルコールは酸化されにくい | C(-OH)にH0 | 実行不可の解説を表示 |
| 脱離 | 分子内脱水 | C-C(-OH)、隣接CにH | アルケン + H₂O |
| 縮合 | 分子間脱水（エーテル生成） | アルコール×2 | R-O-R' + H₂O |
| 縮合 | エステル化 | -COOH と -OH（別分子） | エステル + H₂O |
| 加水分解 | けん化（簡易） | エステル結合 | カルボン酸(塩) + アルコール |
| 付加 | X₂・HX・H₂O・H₂ の付加 | C=C / C≡C | 対応する付加体 |
| 置換 | ベンゼン環の置換（ニトロ化等） | 芳香環のC-H | 置換体（既存機構と連携） |

- 試薬（Br₂・HBr・H₂O・H₂・濃硫酸…）は**内蔵の試薬パレット**から選ぶ（毎回作図させない）。
  二分子反応（エステル化など）は、キャンバス上の別分子を相手として選択する
- マルコフニコフ則など位置選択性は M1〜M3 では扱わず、対称な付加から始める。
  非対称付加は「主生成物のみ・解説付き」で M5 以降に検討

## 3. アーキテクチャ

### 3.1 ファイル構成

- **reactor.js（新設）**: 反応ルールベース＋実行エンジン＋UIカード。DOM操作はここに集約
- **chemistry.js**: 官能基パターン検出の純粋関数を追加（DOM非依存を維持）
  - `findFunctionalGroups(mol)` → `[{type: 'primary_alcohol'|'aldehyde'|'carboxyl'|'ester'|'cc_double'|..., atoms: [...]}]`
  - 既存の isSp3Carbon / 橋検出 / findAromaticBondKeys を流用
- **game.js**: モード結線と分子呼び出しUIのみ（既存パターン踏襲）

### 3.2 反応ルールの表現

```js
{
  id: 'oxidize_primary_alcohol',
  name: '酸化（1級アルコール → アルデヒド）',
  category: '酸化',
  reagentLabel: '酸化剤 [O]',
  // 適用可否: 対象分子（と相手分子）から適用箇所を列挙して返す
  detect(mol) { return sites; },       // 例: [-OH炭素のid, ...]
  // 実行: 分子グラフを書き換える（座標は最小変更の原則）
  apply(mol, site) { ...; return { caption, byproducts: ['H₂O'] }; },
  mechanismName: null,                  // reactions.json に対応機構があれば名前
}
```

- ルールは配列で登録し、「この分子の反応」カードは `detect` が非空のものだけを列挙する
- `apply` は**元の原子座標をできる限り動かさない**（開発方針「環⇔鎖の切替で既存原子を
  動かさない」と同じ思想）。脱離する原子（H₂Oなど）は分子の外側へ平行移動して残す
  （反応機構データと同じ「原子は消さない」原則）か、副生成物として小さく表示する

### 3.3 実行と履歴

- 反応実行は通常の編集と同じく `saveState()` を積んでから行う（**Undo/Redoがそのまま効く**）
- 実行後は updateDrawing → 名称判定カードが自動更新され、生成物名が表示される
  （例: 酸化後に「アセトアルデヒド」）。この既存機能が答え合わせを兼ねる

### 3.4 アニメーション

- **簡易版**: 反応前後の2状態を reaction.js の状態補間（原子は線形移動・結合はフェード）で
  1秒モーフィング。reaction.js の補間ロジックを関数として切り出して共用する
- **機構版**: `mechanismName` があれば反応機構ビューアをその機構で開く（登録データの再生）。
  ユーザー分子への巻矢印の自動付与は将来課題（M5以降）とし、当面は
  「あなたの分子で起きたことを、代表例の機構で見る」体験とする

## 4. 分子の名称呼び出し（ライブラリから配置）

- 「この分子の反応」カードに検索ボックス（datalist）を設置。compounds.json + stages.json の
  名称ライブラリから選ぶと、キャンバスの空き領域（既存分子と重ならない位置）に配置する
- 配置は既存の createTargetFromData → 平行移動 → userMolecule へマージ。1操作=1 Undo
- これにより「エタノールを呼び出す → 酢酸を呼び出す → エステル化」が作図なしで試せる

## 5. UI

- 右パネルに新カード **「⚗ この分子の反応」**（化合物名カードの直下）:
  - 上段: プロパティ表示（P9-3連携）例:「1級アルコール／分子式 C₂H₆O」
  - 中段: 適用可能な反応のボタン列（分類アイコン＋名前＋生成物プレビュー名）
  - 下段: 分子の名称呼び出し検索ボックス
- 分子が複数あるときは、反応ボタン押下時に対象分子（連結成分）をハイライトで選択
- 表示切替（簡易版⇔機構版）はカード内のトグル。既定は簡易版
- パズルモードの正解判定とは独立（ステージのお題とは無関係に遊べる砂場）。
  反応実行モード中も通常編集は可能（実行はあくまで1回の編集操作）

## 6. マイルストーン

- **M1**: 名称呼び出し（検索→空き領域に配置）＋「この分子の反応」カードの骨組み
  （プロパティ表示: アルコールの級・アルデヒド・カルボン酸・C=C の検出まで）
- **M2**: 単分子反応（1級/2級アルコール酸化・アルデヒド酸化・分子内脱水）＋簡易モーフィング
- **M3**: 二分子反応（エステル化・分子間脱水）＋相手分子の選択UI
- **M4**: 試薬パレットと付加反応（Br₂/HBr/H₂O/H₂）・けん化。機構版切替（該当機構へのジャンプ）
- **M5**: 置換（芳香族）・非対称付加の主生成物・ユーザー分子への巻矢印自動付与の検討

## 7. リスク・未決事項

- **生成物レイアウト**: 結合組み換えで原子が重なる場合の退避方向の決め方
  （まずは既存の MIN_CLEARANCE 検査＋外向き平行移動で対応し、監査ページで検証）
- **「酸化」の表現**: 酸化剤は [O] として抽象化（高校教科書の書き方）。
  機構は扱わない（DEVELOPMENT.md P8-7 の保留判断と整合）
- **複数適用箇所**: ジオールなど適用箇所が複数ある場合は、箇所をハイライトして選択させる
- **級の判定と教育**: プロパティ表示から P9-3 の学習リンク（異性体列挙など）へ将来接続する

## 8. P12-5: 反応の前後比較と機構ジャンプ・簡易モーフィング（2026-07-24 設計）

ユーザー要望: **反応を実行すると生成物に置き換わるだけで、前後の比較ができず、
機構もわからない**。3.4節の構想（簡易モーフィング・機構版）を、確実に届く順で2段階に分けて実装する。

### 8.1 第1弾: 前後比較ビュー＋機構ジャンプ

**直近反応の記録**
- `Reactor.execute()` が成功したとき `this.lastReaction = { ruleId, label, before, after }` を保持。
  `before`/`after` は対象分子の **target形式スナップショット（独立コピー・原子ID付き）**。
  反応対象の連結成分だけでなく**キャンバス全体**を写す（分子間反応・副生成物も比較に含める）。
- 保持は直近1件のみ。**次の反応実行で上書き、全消去・モード離脱で破棄**。
  Undo とは独立（↩で戻しても記録は残り、比較は見られる）。

**前後比較ビュー**
- 反応実行後、「⚗ この分子の反応」カードに **「🔍 反応の前後を見る」** ボタンを表示
  （`lastReaction` があるときのみ）。
- 押すとキャンバス領域に **`#rx-compare-overlay`** を重ねて表示（異性体練習の
  `#ip-review-overlay` と同じ流儀。CSSは共通クラス `.canvas-overlay` に括り出して共用）。
  - 「反応前」「反応後」の2図を並置。ヘッダに反応名（例: 分子内脱水）。
  - 図サイズ切替（小/中/大）は `IP_REVIEW_SCALES` を共用。
  - 「← 描画に戻る」で閉じる。図をクリックしても閉じる（異性体練習と同じ操作感）。
- **差分ハイライト**（原子IDの対応付けで機械的に判定。reactor は既存原子のIDを保持し、
  追加原子には新IDを振るため、対応付けは ID 一致だけで取れる）:
  - 消えた結合・次数が下がった結合 → **反応前**の図でオレンジ点線
  - 生成した結合・次数が上がった結合 → **反応後**の図でシアン強調
  - 脱離した原子（前にあって後に無い）→ 反応前の図でオレンジ枠
  - 付加した原子（後にだけある）→ 反応後の図で緑枠
  - 要約文の自動生成（「H₂Oが脱離」等）は第1弾ではやらない（ラベルは反応名のみ）

**機構ジャンプ**
- **reactions.json の各機構に安定ID `id` を新設**（現状は name のみで照合が壊れやすい。
  既存UIは name 表示のままなので互換性に影響なし）。
- reactor の各ルール定義に `mechanismId` を追加。対応表（初期値）:

  | reactor ルール | mechanismId（reactions.json） |
  |---|---|
  | add_br2 | ethene_br2（エテンへの臭素の付加） |
  | add_water | ethene_h2o（エテンへの水付加） |
  | dehydration_intra | ethanol_e1（エタノールの分子内脱水） |
  | esterification | esterification（酢酸とエタノールのエステル化） |
  | hydrolysis_ester | saponification（酢酸エチルのけん化）※加水分解の代表として |
  | aromatic_nitration | benzene_nitration |
  | aromatic_sulfonation | benzene_sulfonation |
  | aromatic_halogenation | benzene_chlorination |
  | oxidize_* / add_h2 / add_hbr / dehydration_inter / acetylation_anhydride | なし（P12-4 の機構追加と連動して随時埋める） |

- `mechanismId` のあるルールは、比較オーバーレイと反応カードに
  **「⚗ この反応の機構を見る（代表例）」** ボタンを表示。押すと
  `setMode('learn')` → 反応機構ビューアで該当機構を選択・再生開始。
  注記を添える: **「※ あなたの分子そのものではなく、代表例の分子で機構を再生します」**
  （ユーザー分子への巻矢印自動付与は M5 の将来課題のまま）。
- 分子はモードをまたいで保持される（P10仕様）ので、見終えたら自由モードに戻れば続きから編集できる。

**テスト（第1弾）**
- RX1: エタノール→「酸化」実行→前後比較を開くと2図が描画され、
  反応前=エタノール・反応後=アセトアルデヒドのトポロジーで、差分ハイライトが存在する
- RX2: mechanismId のあるルール実行→「機構を見る」→ learnモードに切替わり、
  ビューアが対応機構をロードしている
- RX3: 全ルールの mechanismId が reactions.json の id に実在する（データ整合の機械検証）。
  reactions.json の id 重複なしも検証

### 8.2 第2弾: 簡易モーフィング（実行時アニメ）

- `execute()` の適用後、生成物への切替を**約1秒のモーフィング**で見せる:
  共通原子（ID一致）は座標を線形補間、消える原子・結合はフェードアウト、
  現れる原子・結合はフェードイン、次数変化はクロスフェード。
- 実装: reaction.js の状態補間・描画ロジックから共通関数を切り出して共用
  （3.4節の方針どおり）。アニメは一時レイヤに描き、完了時に通常描画へ戻す。
- **分子データは実行時点で即確定**させ、アニメは表示のみ（検証・Undo・監査に一切影響しない
  ＝「検証はトポロジーのみ」の原則に整合）。
- 再生中のキャンバス入力は無視（最長1秒）。タップでスキップ即完了。
  `prefers-reduced-motion` 環境ではアニメせず即切替。
- テスト: 補間関数の純関数テスト（t=0で反応前・t=1で反応後に一致）＋
  実行後の最終分子がアニメ無し時と同一（分子式・正準コード）。見た目は手動確認。
